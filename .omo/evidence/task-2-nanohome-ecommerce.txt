## Unit tests  — PASS (with caveat)

Command (literal from spec): `bunx vitest run src/lib/env.test.ts`

Result: FAILS at the runner level, NOT at the test logic. Pre-existing
environment issue (scaffolded in T1, unrelated to T2):

    [plugin: externalize-deps] "@vitejs/plugin-react" resolved to an ESM file.
    ESM file cannot be loaded by `require`.
    See https://vite.dev/guide/troubleshooting.html#this-package-is-esm-only
    EXIT=1 (config-bundle-time error; env.test.ts body never reached)

Root cause: `vitest.config.ts` (T1) imports `@vitejs/plugin-react` (^6.0.3,
ESM-only) and Vite's esbuild-based config bundler externalizes plugin deps
through `require`. Vitest 2 + Vite 6 in this toolchain cannot require an ESM
plugin from a CJS-style config bundle. This is a T1/test-runner tooling issue,
not a T2 deliverable.

Workaround used to exercise env.test.ts logic: a minimal temp vitest config
(without the React plugin, node environment) placed at
`vitest.minimal.config.ts` in the project root for the duration of the run:

    bunx vitest run --config vitest.minimal.config.ts src/lib/env.test.ts

Observable:

    ✓ src/lib/env.test.ts (4 tests) 20ms

Temp config `vitest.minimal.config.ts` deleted after the run. `git status`

## Risks / notes

1. `bunx vitest run src/lib/env.test.ts` (the literal spec command) does not
   run due to a pre-existing T1 toolchain issue: `@vitejs/plugin-react` is
   ESM-only and Vite's esbuild config bundler cannot load it via `require`.
   The test logic itself passes 4/4 with a minimal vitest config (above). Fix
   would be in `vitest.config.ts` (T1 scope), not T2.

2. `next.config.ts` now has a side-effect `import "./src/lib/env";` for
   build-time env validation. This means every Next CLI invocation
   (`build`, `dev`, `start`, `lint` if it touches config) requires the four
   mandatory env vars (`NEXT_PUBLIC_SUPABASE_URL`,
   `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`, `SUPABASE_SERVICE_ROLE_KEY`,
   `CRON_SECRET`). Developers must populate `.env.local` before `next dev`.
   `.env.example` documents all four as required.

3. `instrumentation.ts` (scaffolded in T1) still reads `process.env.AXIOM_TOKEN`
   and `process.env.AXIOM_DATASET` directly, NOT through the validated `env`
   object. T2's schema only lists `NEXT_PUBLIC_AXIOM_DATASET` /
   `NEXT_PUBLIC_AXIOM_TOKEN` (public). The mismatch between T1's
   `instrumentation.ts` and T2's schema for Axiom vars is a T1/T15 concern, not
   T2.

4. The ESLint path-based allowlist (`src/lib/supabase/**`, `src/app/api/**`,
   `**/route.*`, `**/actions.*`) is a heuristic for "server-only entry
   points" because ESLint flat config cannot match on `"use client"`
   directive content. Any new server-only entry-point shape not matching the
   allowlist patterns would fall under the restriction and need either an
   allowlist update or a different restriction strategy.
