  - PostgREST `.or()` filter operands now quote search terms containing reserved grammar characters before interpolation.
  - Literal double quotes and backslashes inside quoted filter operands are backslash-escaped.
  - Product visibility filters: `validated=true`, `approved=true`.
  - Variant visibility filters in embedded contract: `variants.validated=true`, `variants.approved=true`.
  - Variant match prequery on `variants` with `&@~` across `name`, `sku`, `finish`, `finish_vi`, filtered to visible variants with non-null `product_id`.
  - Product query uses PGroonga `&@~` across locale-prioritized product fields and includes product IDs matched by variants.
  - Vietnamese field priority: product `name_vi`, `description_vi`; variant `finish_vi` first.
  - English field priority: product `name`, `description`; variant `name`, `sku`, `finish` first.
  - Cross-locale matches remain included at lower priority order in the filter lists.
  - Pagination preserved with `productRange()` and `.range(from, to)`.
- `src/lib/queries/search.test.ts`
  - Added focused Supabase query-builder mock tests for empty query, visibility filters, PGroonga operator strings, locale-prioritized cross-locale matching, variant search, ordering, and pagination.

Narrowed contract:
- Backend-only T16 implemented products + variants only, no news.
- Variant search is implemented as a typed prequery for visible variant `product_id` values, then product retrieval by product-field PGroonga OR matched variant product IDs. This avoids risky cross-table embedded OR syntax in Supabase/PostgREST while preserving a testable product-returning contract.
- Locale-aware ranking is represented by deterministic preferred-field ordering in the PGroonga disjunction plus locale-specific `.order(name_vi|name)`, not a raw SQL rank expression, because no RPC/schema change was requested and no Supabase remote mutation tools are allowed.
